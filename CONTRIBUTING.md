## Development Setup

1. **Fork the repository**
   ```bash
   # Fork on GitHub, then clone your fork
   git clone https://github.com/your-username/context-mesh.git
   cd context-mesh
   ```

2. **Create a branch**
   ```bash
   git checkout -b feature/your-feature-name
   # or
   git checkout -b fix/your-bug-fix
   ```

3. **Make your changes**

4. **Test your changes**
   - Verify documentation renders correctly
   - Test examples if you modified them
   - Check for typos and broken links

5. **Commit your changes**
   ```bash
   git add .
   git commit -m "Description of your changes"
   ```

6. **Push to your fork**
   ```bash
   git push origin feature/your-feature-name
   ```

7. **Create a Pull Request**
   - Use the PR template
   - Link related issues
   - Describe your changes clearly

## Contribution Workflow

### For Code Changes

1. **Create an issue** (if one doesn't exist)
   - Describe what you want to change
   - Get feedback before starting work

2. **Follow Context Mesh principles**
   - Document intent before implementation
   - Update context files if needed
   - Follow existing patterns

3. **Write clear commit messages**
   ```
   type: Short description

   Longer description if needed
   ```
   
   Types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`

4. **Update documentation**
   - If you add features, update relevant docs
   - Update examples if they're affected
   - Update CHANGELOG.md

### For Documentation Changes

1. **Identify what needs improvement**
   - Unclear explanations
   - Missing information
   - Outdated content

2. **Make the change**
   - Keep the same style and tone
   - Add examples if helpful
   - Update related sections

3. **Test the documentation**
   - Check markdown rendering
   - Verify links work
   - Test code examples

## Context Mesh Guidelines

When contributing, please follow Context Mesh principles:

### 1. Context as Primary Artifact

- Document **why** decisions were made
- Update context files when making changes
- Preserve rationale and alternatives

### 2. Intent-Driven

- Document intent before implementation
- Update feature intents if adding features
- Create decisions for technical choices

### 3. Living Documentation

- Keep documentation current
- Update examples to reflect changes
- Document learnings and outcomes

### 4. Simplicity

- Keep changes focused and clear
- Avoid over-engineering
- Follow existing patterns

## Pull Request Process

1. **Use the PR template**
   - Fill out all relevant sections
   - Check Context Mesh related items
   - Link related issues

2. **Keep PRs focused**
   - One feature or fix per PR
   - Keep changes small when possible
   - Split large changes into multiple PRs

3. **Update documentation**
   - If adding features, update docs
   - Update examples if needed
   - Update CHANGELOG.md

4. **Respond to feedback**
   - Address review comments
   - Make requested changes
   - Ask questions if unclear

5. **Wait for review**
   - Maintainers will review your PR
   - Be patient and responsive
   - Make changes as requested
